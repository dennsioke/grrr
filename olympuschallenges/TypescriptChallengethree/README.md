Title
Scroll-progress keyframes for multi-stop parallax animations

Add a keyframes prop that accepts an array of stops -- each with a progress (0-1) and any mix of the existing effect properties -- so that values interpolate piecewise between adjacent stops as the element scrolls. When a stop omits an effect property that appeared in a previous stop, the last known value carries forward so that the effect holds steady through that segment. Stops after the first can carry an easing property as a four-number bezier tuple that controls the curve into that stop; the first stop must not have an easing. Easing values must follow standard cubic-bezier rules (x1 and x2 in [0,1], exactly four finite numbers). Use the `bezier-easing` npm package to evaluate cubic-bezier curves -- do not implement a custom solver. The array must start at 0 and end at 1, be ordered by ascending progress with every stop's progress in [0,1], and throw if combined with individual effect props or speed. All effect values must be finite, and units for the same effect must be consistent across stops. Banner layers with keyframes and expanded enabled need to compute their negative top/bottom overflow margins from the largest positive and negative translateY across all stops rather than just two endpoints.
Inside useParallax, when keyframes are present, wire the keyframe styles to scroll updates by setting an onProgressChange callback on the parallax controller element props. This callback receives the current scroll progress (0-1) and must apply the resolved keyframe styles (transform and opacity) directly to the target DOM element. When keyframes are not provided, useParallax must behave identically to the existing implementation.
The solution must export a validateKeyframes(keyframes, props) function and a resolveKeyframeStyles(keyframes, progress) function from src/helpers/resolveKeyframeStyles.ts. resolveKeyframeStyles accepts an array of keyframe stops and a numeric progress value in [0,1] and returns a plain object with a transform string and an opacity string. The transform string contains one function call per active transform effect (translateX, translateY, scale, scaleX, rotate) with the interpolated value at the given progress. Numeric translate values use percent units (e.g. translateY(50%)), string values preserve their original unit (e.g. translateY(20px)), scale values are unitless (e.g. scale(1.5)), and rotate values use deg (e.g. rotate(180deg)). Opacity is returned as a plain numeric string (e.g. 0.5). Unit consistency validation across stops must compare actual CSS unit strings -- two stops like translateY: '10px' and translateY: '50%' must throw because the units differ.
Do not regenerate package-lock.json or run npm install with flags that rewrite the lockfile. The Dockerfile handles dependency installation.
